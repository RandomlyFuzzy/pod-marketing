# Agent System

Local multi-agent framework built on the OpenAI Agents SDK, Ollama-compatible models, JSON agent definitions, and MCP tools.

The current goal is a local Codex-like setup: a planner receives a task, sees the registered team and tools, delegates to specialists, stores intermediate results, and passes output through a reviewer.

## Current Shape

```text
CLI / MCP client
  |
  v
orchestrator_agent.py
  |-- run planner
  |-- run reviewer
  |-- retry up to a small iteration limit
  v
planner_agent.py or definitions/planner.json
  |-- sees the live team manifest
  |-- chooses registered agents by exact name
  |-- delegates through delegate_to_agent()
  v
base.py
  |-- creates Agents SDK Agent objects
  |-- attaches configured function tools
  |-- attaches configured MCP servers
  v
definitions/*.json
  |-- source of truth for agent names, model choices, instructions, tools
```

## Important Files

| File | Purpose |
|------|---------|
| `registry.py` | Loads agent JSON and builds the live team/tool manifest |
| `base.py` | Creates Agents SDK agents from definitions |
| `planner_agent.py` | Dedicated planner with persistence and delegation tools |
| `reviewer_agent.py` | Quality gate for planner output |
| `orchestrator_agent.py` | Planner/reviewer loop with capped iterations |
| `management_server.py` | Optional MCP server for registry CRUD and delegation |
| `LOCAL_CODEX_DIAGNOSIS.md` | Diagnosis of what was broken and what is still missing |

## Team Manifest

The planner now gets a live manifest containing each registered agent's:

- agent name
- description
- model
- configured function tools
- configured MCP tools

Print it with:

```bash
./start-agent.sh list
```

The manifest is generated by `registry.team_manifest()`. Do not maintain a separate hardcoded agent list.

## CLI Usage

```bash
# Show registered agents, models, and effective tools
./start-agent.sh list

# Run the full planner/reviewer orchestrator
./start-agent.sh orchestrator "Research current POD niches and assess profitability"

# Bare goal text is also routed through the orchestrator
./start-agent.sh "Research current POD niches and assess profitability"

# Legacy direct-agent shapes are converted into orchestrator goals
./start-agent.sh researcher "Find current 3D printed item market data"
./start-agent.sh general researcher "Find current 3D printed item market data"
```

## Agent Definitions

Agents live in `agent_system/definitions/*.json`.

Minimal example:

```json
{
  "name": "my_agent",
  "model": "planner-model",
  "description": "What this agent does",
  "instructions": "You are a specialist. Do the task and return concise evidence.",
  "mcp_servers": ["scrapling"],
  "tools": [
    "brain_search",
    "brain_save_note",
    "brain_track_concept",
    "brain_publish_summary"
  ]
}
```

If `tools` is omitted, the registry falls back to core function tools for compatibility. If `mcp_servers` is omitted, it assumes Scrapling.

## Tooling

Configured function tools include:

- `check_trends`
- `validate_with_trends`
- `delegate_to_agent`
- `create_agent_type`
- `brain_save_note`
- `brain_track_concept`
- `brain_search`
- `brain_publish_summary`

Configured Scrapling MCP tools are represented in the manifest as:

- `get`
- `fetch`
- `stealthy_fetch`
- `bulk_get`
- `bulk_fetch`
- `screenshot`

Scrapling must be installed and available as:

```bash
scrapling mcp
```

## Known Limitation

The remaining blocker is model/tool-call reliability. If the active local model writes what it intends to do instead of issuing OpenAI-compatible tool calls, the planner will still fail at execution time. That is a model/controller problem, not an agent registry problem.

The next meaningful refinement is a deterministic controller that rejects planner outputs when an execution task completed with zero tool calls.
